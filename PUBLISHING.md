# Publishing gacli to npm

Releases are **fully automated** by [semantic-release](https://github.com/semantic-release/semantic-release)
on every push to `main`. Authentication to npm uses **OIDC Trusted Publishing**
(no `NPM_TOKEN`, no secrets in CI). The npm registry verifies the GitHub OIDC
token directly against a trusted-publisher relationship configured on
npmjs.com.

Why OIDC? **npm revoked all classic Automation tokens on 2025-11-19.** Granular
tokens require an IP allowlist for 2FA bypass — unworkable on GitHub-hosted
runners' elastic IP pool. OIDC is the only viable path.

## What's already decided

| Decision | Value | Where |
|---|---|---|
| Package name | `@nalyk/gacli` | `package.json:name` |
| License | MIT | `LICENSE` + `package.json:license` |
| Bin name | `gacli` (lives at `~/.local/bin/gacli` after install) | `package.json:bin` |
| Version source of truth | `package.json` (read at runtime by `src/index.ts`) | `src/index.ts` |
| Versioning | Conventional Commits → semantic-release | `release.config.js` |
| Publish trigger | Push to `main` (semantic-release decides if it warrants a release) | `.github/workflows/release.yml` |
| Authentication | GitHub OIDC → npm trusted publisher (no token) | release.yml + npm UI |
| Provenance | Auto-generated by npm when publishing via OIDC | implicit |
| Stable dist-tag | `latest` (releases from `main`) | `release.config.js:branches` |
| Pre-release dist-tags | `next` (from `next` branch), `beta` (from `beta` branch) | `release.config.js:branches` |
| GitHub deployment env | `npm` (OIDC environment claim must match) | release.yml `environment.name` |

## Day-to-day workflow

You commit. Everything else is automatic.

```bash
git commit -m "feat(skills): add auto-detect of installed agents"
git push origin main
```

The `release.yml` workflow runs on the push, `pnpm verify` passes, semantic-release
inspects commits since the last tag, decides this is a `feat:` so the next version
is a minor bump (e.g. `1.1.0` → `1.2.0`), bumps `package.json`, prepends
`CHANGELOG.md`, commits both back to `main` as `chore(release): v1.2.0 [skip ci]`,
tags `v1.2.0`, pushes the tag, publishes to npm via OIDC, and creates a GitHub
Release with auto-generated notes + the `.tgz` asset.

The `[skip ci]` marker stops the workflow from re-triggering on its own commit.

### Commit-prefix → release-type cheatsheet

| Prefix | Release | Example |
|---|---|---|
| `feat: …` | minor | `feat(skills): add --no-detect flag` |
| `fix: …` | patch | `fix(install): handle EXDEV on cross-fs rename` |
| `perf: …` | patch | `perf(report): cache property metadata` |
| `refactor: …` | patch | `refactor(skills): consolidate per-target factory` |
| `docs(readme): …` | patch | `docs(readme): document --force flag` |
| `chore: …`, `docs: …`, `test: …`, `ci: …`, `build: …`, `style: …` | none | `chore: bump deps` |
| any commit body containing `BREAKING CHANGE:` | major | `feat!: rename --account to --account-id` |

Use `!` after the type to flag a breaking change inline (`feat!:` triggers a
major bump). Or include `BREAKING CHANGE: <description>` in the commit body.

## Pre-release channels

Push to a `next` or `beta` branch instead of `main`. semantic-release publishes
`X.Y.Z-next.N` or `X.Y.Z-beta.N` under the corresponding npm dist-tag, and the
GitHub Release is marked as a prerelease.

```bash
git checkout -b next
git commit -m "feat: experimental thing"
git push origin next
# → @nalyk/gacli@1.3.0-next.1 on the @next dist-tag
```

Consumers test with `npm install -g @nalyk/gacli@next`. When you merge `next`
into `main`, semantic-release promotes it to `latest`.

## What's verified at publish time

- `pnpm verify` — lint + type-check + 83 tests + skill-lint + build, all green or no publish.
- The new tarball is `npm install -g`-tested locally before semantic-release uploads.
- After publish, the registry version is reinstalled from the public registry to
  verify a real consumer's install path actually works.
- npm provenance attestation is auto-generated by OIDC trusted publishing; consumers
  run `npm audit signatures` to verify the package was built from this repo at this
  commit by GitHub Actions.

## What is shipped (the `files` allowlist)

Only:

- `dist/` — compiled JS + `.d.ts`
- `extensions/` — bundled AI-CLI skill packages
- `README.md`, `help.md` — user-facing docs
- `LICENSE`

NOT shipped: `src/`, `test/`, `.serena/`, `.github/`, `.claude/`, configs, lockfiles.

## One-time setup (already done)

OIDC trusted publishing is wired up at
<https://www.npmjs.com/package/@nalyk/gacli/access> (Trusted Publishers tab).
The relationship is keyed on **workflow filename = `release.yml`** and
**environment = `npm`** — renaming the workflow file would break publishing
until the npm UI is updated to match.

## Branch protection requirement

semantic-release pushes the bump commit + tag back to `main`. The `main` branch
protection rule must allow GitHub Actions / repository administrators to bypass
required PRs and required status checks. Without that, the
`@semantic-release/git` step fails with HTTP 403.

Settings → Branches → main rule → "Allow specified actors to bypass required
pull requests" → add the bot or grant admin bypass.

## Manual fallback

If the trusted publisher relationship breaks (e.g., you renamed the workflow file
or changed the environment name), publish from your machine with an OTP:

```bash
git checkout main && git pull
pnpm install --frozen-lockfile
pnpm verify
pnpm pack
npm publish *.tgz --otp=XXXXXX --access public
```

Or trigger semantic-release locally (it will use your local `npm whoami` session):

```bash
GITHUB_TOKEN=$(gh auth token) pnpm exec semantic-release --no-ci
```

Then create the matching git tag + GitHub Release manually if semantic-release
didn't.
