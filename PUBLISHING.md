# Publishing gacli to npm

The repo publishes to npm via **OIDC Trusted Publishing** (May 2026 standard) — no
`NPM_TOKEN`, no secrets in CI. The npm registry verifies the GitHub OIDC token directly
against a trusted-publisher relationship configured on npmjs.com.

Why not classic tokens? **npm revoked all classic Automation tokens on 2025-11-19.**
Granular tokens require an IP allowlist for 2FA bypass — unworkable on GitHub-hosted
runners' elastic IP pool. OIDC is the only real path forward.

| Decision | Value | Where |

| License | MIT | `LICENSE` + `package.json:license` |
| Bin name | `gacli` (lives at `~/.local/bin/gacli` after install) | `package.json:bin` |
| First version to publish | `1.0.1` | `package.json:version` |
| Publish trigger | Tag push matching `v*.*.*` | `.github/workflows/release.yml` |
| Authentication | GitHub OIDC → npm trusted publisher (no token) | release.yml + npm UI |
| Provenance | Auto-generated by npm when publishing via OIDC | implicit |
| Stable dist-tag | `latest` for `vX.Y.Z` | release.yml prerelease detection |
| Pre-release dist-tag | `next` for `vX.Y.Z-anything` | release.yml prerelease detection |
| GitHub deployment env | `npm` (OIDC environment claim must match) | release.yml `environment.name` |

## One-time setup (chicken-and-egg)

OIDC trusted publishing **requires the package to already exist on npm** before the
trusted-publisher relationship can be configured. So the very first publish must be done
manually with a one-time password.

### Phase 2 — Bootstrap publish (HUMAN, one-time)

```bash
# 1. Make sure you're on main with the OIDC-ready package.json
git checkout main && git pull

# 2. Confirm logged-in identity (web flow if you haven't logged in yet)
npm whoami        # if missing: `npm login`

# 3. Get a fresh OTP from your authenticator app and publish IMMEDIATELY.
#    Don't wait for npm to prompt — npm 11.x sometimes 403s on 2FA-on-write
#    accounts instead of prompting. Pass --otp explicitly.
npm publish --otp=XXXXXX --access public

# 4. Verify the package is live
npm view @nalyk/gacli@1.0.1 dist.tarball
npm view @nalyk/gacli bin
```

If the publish 403s with "You may not perform that action with these credentials":
`npm logout && npm login`, refresh OTP, retry. If it persists, your npm account has
unusual lockdowns that need to be resolved with npm support.

### Phase 3 — Configure trusted publisher (HUMAN, one-time)

Once `@nalyk/gacli` exists on npm:

1. Go to https://www.npmjs.com/package/@nalyk/gacli/access
2. **Trusted Publishers** tab → **Add trusted publisher**
3. Fill in:

| Field | Value |
|---|---|
| Provider | GitHub Actions |
| Organization or user | `nalyk` |
| Repository | `gacli` |
| Workflow filename | `release.yml` (filename only, **not** the full path) |
| Environment name | `npm` |

4. Save. No token, no further config.

### Phase 4 — Verify OIDC works (AUTOMATED)

Tell me you've finished Phases 2 and 3, and I'll dispatch the workflow on `main` (no tag).
The workflow's idempotency guard sees the version already on npm and skips publish; the
release step is skipped because it's not a tag run; the green run end-to-end is proof
that OIDC is wired correctly. Subsequent `v*.*.*` tag pushes will publish via OIDC with
zero secrets.

## Ongoing release flow

After the bootstrap, every release is one command pair:

```bash
pnpm version patch                       # bumps package.json + creates v* tag
git push origin main --follow-tags       # ships to npm + creates GitHub Release
```

Use `minor` or `major` for SemVer-appropriate bumps. The workflow's tag↔version guard
catches mismatches before publishing; the idempotency guard catches accidental re-pushes
of an existing version.

## Pre-releases

```bash
pnpm version prepatch --preid=beta       # 1.0.1 → 1.0.2-beta.0
git push origin main --follow-tags
```

The workflow detects the `-` and publishes under dist-tag `next` instead of `latest`,
and marks the GitHub Release as prerelease.

## What's verified at publish time

- `pnpm verify` — lint + type-check + test + build all green or no publish.
- Tag↔version equality — the workflow fails if `git tag v1.2.3` is pushed while
  `package.json:version` is `1.2.4`.
- Local pack is `npm install -g`-tested before upload.
- After publish, the registry version is reinstalled from the public registry to
  verify a real consumer's install path actually works.
- npm provenance attestation is auto-generated by OIDC trusted publishing; consumers
  run `npm audit signatures` to verify the package was built from this repo at this
  commit by GitHub Actions.

## What is shipped (the `files` allowlist)

Only:

- `dist/` — compiled JS + `.d.ts`
- `README.md`, `help.md` — user-facing docs
- `LICENSE`

NOT shipped: `src/`, `test/`, `.serena/`, `.github/`, `.claude/`, configs, lockfiles.

## Manual fallback

If the trusted publisher relationship breaks (e.g., you renamed the workflow file or
changed the environment name), publish from your machine with an OTP:

```bash
pnpm verify
npm publish --otp=XXXXXX --access public
```

Then re-do Phase 3 with the new workflow filename / environment name.

## Unpublishing

You have **72 hours** after publish to `npm unpublish @nalyk/gacli@<version>`. After
that the version is permanent. If you ship a broken release: bump the patch and
publish the fix, don't try to unpublish.

## Adding collaborators with publish access

```bash
npm access grant read-write @nalyk:developers @nalyk/gacli
# or per-user:
npm access grant read-write <username> @nalyk/gacli
```

You retain admin via your scope ownership.
